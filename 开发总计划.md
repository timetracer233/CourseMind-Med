# 医教精华压缩 Agent 开发总计划

## 1. 战略结论

本项目选择“方案一：医教精华压缩 Agent”，但执行策略必须收窄：

**不追求 5 小时内真实完整处理 7 本 800MB 医学教材；追求评委打开公网链接后，能顺畅完成一条可信的教材整合演示路径。**

比赛获胜点不是功能堆满，而是：

1. 页面能用。
2. P0 核心路径能跑。
3. 文档完整且对齐赛题。
4. RAG 有引用。
5. 图谱可交互。
6. 跨教材整合有决策、有压缩比、有解释。

医学教材是赛方测试案例；系统表述为“通用教材知识整合智能体”，但开发优先让 1-2 本医学教材小样例稳定跑通。

## 2. 已定决策

| 决策项 | 选择 | 说明 |
| --- | --- | --- |
| 项目方向 | 医教精华压缩 Agent | 最贴赛题 |
| 领域优先级 | 医学教材优先，架构通用 | 先打穿测试数据 |
| UI | Gradio | 5 小时最快 |
| 大模型 | DeepSeek | 用户已确定 |
| 部署 | 魔搭创空间优先，Gradio share 保底 | 先本机跑通，再部署 |
| 图谱 | pyvis HTML | 快速实现可交互图谱 |
| 检索 | sklearn TF-IDF / sentence-transformers 二选一 | 优先稳定，避免依赖翻车 |
| 教师反馈 | 简化版规则指令 | 满足验收，不做复杂 NLP |
| 文档 | 与代码并行 | 文档 + Agent 架构约 35 分，不能最后补 |

## 3. 必须砍掉的内容

为了 5 小时内不翻车，以下内容不进入第一版：

- DOCX / Excel 支持。
- Docker / docker-compose。
- 混合检索 / Rerank。
- 多视图图谱。
- 本地大模型。
- 复杂 LangGraph / CrewAI 多 Agent。
- 全量 7 本大 PDF 深度抽取。
- 扫描版 PDF OCR。

这些可以写为“后续增强”，不要在代码第一版硬做。

## 4. 第一版必须完成的 P0 简化闭环

| 功能 | 实现标准 |
| --- | --- |
| 文件上传解析 | PDF / MD / TXT 可上传，可显示章节和字数 |
| 知识点抽取 | DeepSeek 抽取；失败时规则兜底 |
| 图谱展示 | pyvis 可缩放、拖拽、点击/悬停 |
| RAG 问答 | 检索 top-k chunk，答案带教材/章节/页码引用 |
| 跨教材整合 | 2 本教材可生成 merge/keep/remove 决策 |
| 压缩比 | 显示原始字数、整合后估算字数、压缩比 <= 30% |
| 教师反馈 | 支持“保留/删除/不要合并/为什么合并”四类简化指令 |
| 整合报告 | 可生成 `report/整合报告.md` |
| 开发文档 | README + 需求分析 + 系统设计 + Agent 架构说明 |
| 公网访问 | 魔搭创空间或 Gradio share 至少一个可用 |

## 5. 文档与代码并行机制

文档不能放到最后 1 小时。

开发从一开始就保留两条线：

### 主线 A：代码闭环

- Gradio 页面。
- 解析。
- RAG。
- 图谱。
- 整合。
- 反馈。
- 报告。

### 主线 B：文档闭环

- README。
- `docs/需求分析.md`
- `docs/系统设计.md`
- `docs/Agent架构说明.md`
- `report/整合报告.md`

文档先写“设计版”，每完成一个功能就回填真实数据、截图描述、实际限制。这样最后 30 分钟只做校验，不临时写 5 份文档。

## 7. 黄金演示路径

必须提前准备一条稳定演示路径，评委看到的是这条路径，不是系统极限能力。

### 演示输入

- 1-2 本小型医学教材样例。
- 每本控制在几十页或若干章节。
- 若正式 7 本教材过大，演示时使用快速模式处理前 N 页或前 N 章。

### 演示步骤

1. 上传 2 个教材文件。
2. 点击“解析教材”，看到章节数、字数、状态。
3. 点击“构建知识图谱”，看到可拖拽图谱。
4. 点击“跨教材整合”，看到合并决策和压缩比。
5. 输入预设问题，看到带引用的 RAG 答案。
6. 在教师反馈里输入“不要合并 炎症 和 炎症反应”或类似指令。
7. 生成整合报告。

### 推荐预设问题

根据样例教材实际内容准备 3 个：

- “炎症的定义是什么？”
- “免疫应答包含哪些核心环节？”
- “动作电位和静息电位有什么关系？”

问题必须能在样例中找到原文来源。

## 9. 魔搭创空间部署策略

推荐流程：

1. 本机先写好并跑通。
2. 推送 GitHub。
3. 魔搭创空间创建 Gradio 应用。
4. 配置 `DEEPSEEK_API_KEY` 环境变量。
5. 启动并测试公网链接。

注意：不要把第一次完整调试放到魔搭。魔搭用于最终部署，不是主开发环境。

保底方案：

- 如果魔搭构建失败或来不及，使用 Gradio share：

```python
demo.launch(share=True)
```

拿到临时公网链接后提交。只要链接在评审当天可访问，就能救场。

## 10. 风险控制

| 风险 | 应对 |
| --- | --- |
| 文档来不及 | 已提前生成模板；开发中持续回填 |
| 功能太多 | P1 全砍，只做 P0 简化闭环 |
| PDF 太大 | 快速模式，只处理前 N 页/章节 |
| DeepSeek 不稳定 | 检索式答案兜底；知识抽取规则兜底 |
| 图谱失败 | 节点/边表格兜底 |
| 魔搭失败 | Gradio share 保底 |
| 评委上传陌生文件 | 文件解析失败不崩，显示错误原因 |

## 11. 最终成功标准

比赛提交前必须满足：

- GitHub public。
- 公网链接可打开。
- 页面能完成黄金演示路径。
- RAG 答案有引用。
- 图谱可交互。
- 整合决策和压缩比可见。
- 教师反馈能修改决策。
- 5 份核心文档存在且内容对齐赛题。
- PDF 教材未提交到仓库。

如果只能保一件事：保“公网可访问 + 黄金演示路径完整”。功能朴素没关系，不能打不开、不能无引用、不能无文档。

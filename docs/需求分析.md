# 需求分析

## 1. 背景

高校和职业教育中同一门课程往往同时使用多本教材。以医学为代表的不同教材可能从生理、病理、营养、康复、护理等角度讲解同一知识点，内容既重复又互补。教师备课需要判断重复内容、保留内容和缺口；学生容易被无来源的 AI 回答误导。

本项目解决：用 AI 将多本教材整合为可检索、可解释、可压缩的知识体系，在保留教学完整性的前提下降低重复内容。

**医学教材优先验证，系统架构可迁移到其他学科。**

## 2. 用户与场景

### 2.1 学科教师

场景：备课、课程重构、教材比选。

痛点：
- 多本教材中重复内容多，人工比对成本高
- 不容易判断某一概念在哪本书中讲得更完整
- 删除重复内容时担心破坏教学逻辑

### 2.2 学生

场景：复习、跨教材查漏补缺。

痛点：
- 不知道不同教材中的同一概念是否等价
- 很难快速定位答案来源
- 容易被无来源的 AI 回答误导

### 2.3 教学管理者

场景：课程资源整合、教材体系建设。

痛点：
- 缺少数据化依据判断教材覆盖、重复和缺失
- 缺少面向课程建设的整合报告

## 3. 核心目标

1. 支持上传多本教材并解析为统一结构（PDF / Markdown / TXT）
2. 从教材章节中抽取知识点和关系
3. 为单本和多本教材生成可交互知识图谱
4. 跨教材识别重复、互补和缺失
5. 将整合后内容压缩到原始体量 30% 以内
6. 基于教材内容进行 RAG 问答，并提供原文引用
7. 支持教师通过多轮对话修改整合决策
8. 生成整合报告

## 4. 知识点粒度定义

本项目将知识点定义为教材中可独立教学、可被提问、可与其他知识点建立关系的最小语义单元。

优先抽取以下类型：
- 核心概念：如"炎症""动作电位""免疫应答"
- 机制原理：如"血糖调节机制"
- 方法技术：如"心肺复苏步骤"
- 疾病/现象：如"高血压""细胞坏死"
- 评估指标：如"BMI""血压分级"
- 干预措施：如"运动处方""饮食控制"

粒度控制原则：
- 过粗的章节标题不能直接作为唯一知识点
- 过细的孤立名词如果缺少定义，不优先保留
- 能独立成为问答对象的内容优先保留

## 5. 重复判定标准

两个知识点被判定为重复或可合并，需要满足至少一种条件：

1. 名称完全一致或规范化后高度一致（相似度 ≥ 0.82）
2. 名称不同但定义描述同一概念
3. 中英文术语或别名对应同一概念
4. `name + definition` 的相似度超过阈值
5. LLM 判断二者在教学语义上等价（当前版本未启用，作为后续增强）

不会合并的情况：
- 名称相近但层级不同，例如"免疫"与"体液免疫"
- 一个是过程，一个是结果
- 一个是概念，一个是应用场景
- 教师反馈明确要求拆分

## 6. 压缩比计算方式

赛题要求整合后内容不超过原始体量 30%。

当前版本采用知识点级压缩估算：

```text
原始总字数 = 所有教材解析正文总字数
整合后估算字数 = 原始总字数 - 合并知识点数 × 50（经验值）
压缩比 = 整合后估算字数 / 原始总字数
```

页面展示：
- 原始总字数
- 整合后估算字数
- 压缩比百分比

**当前版本为估算而非正文压缩**，后续增强版本可基于整合结果生成完整精华教材正文。

## 7. 教学连贯性保障

1. 保留前置依赖节点：若 A 是 B 的 prerequisite，删除 B 前不得删除 A（当前版本未实现，作为后续增强）
2. 保留高频核心节点：多本教材共同出现的知识点通常代表课程核心内容
3. 保留互补节点：只在一本教材出现但能补足其他教材缺口的内容标记为 keep
4. 决策可解释：每条 merge/keep/remove 都提供理由和置信度
5. 教师可覆盖：教师反馈优先于自动合并规则

## 8. RAG 分块策略

分块参数：
- chunk_size：700 中文字符
- overlap：100 中文字符
- 检索 top-k：默认 5

选择理由：
- 700 字能容纳一个完整概念解释或教材段落
- 重叠窗口可以减少概念被切断造成的检索缺失
- top-5 能在上下文长度和答案覆盖之间取得平衡

每个 chunk 必须包含：
- 教材名
- 章节名
- 页码
- 原文内容
- 相关度分数

向量化策略：

- 当前 5 小时版本使用 sklearn TF-IDF 做本地轻量向量化。
- 检索使用 cosine similarity 返回 top-k chunk。
- 该方案不依赖在线 Embedding API 或本地大模型下载，优先保证比赛现场稳定演示。
- 后续版本可升级到 API embedding 或本地 sentence-transformers。

## 9. 约束与风险

### 时间约束

比赛只有 5 小时，系统优先实现完整可演示闭环，不追求一次性处理全部超大教材的最优性能。快速模式默认只处理前 60 页或前 8 章。

### 数据约束

7 本教材可能体积较大，不允许提交到 GitHub。系统必须支持前端上传。

### 模型约束

LLM 可能出现 JSON 格式错误、回答幻觉或网络波动。系统保留检索式兜底和异常提示。

### 部署约束

公网链接必须可访问。魔搭创空间为优先部署平台，Gradio share 作为备用。

## 10. MVP 验收标准

- 上传至少 2 本教材（PDF / MD / TXT）
- 解析章节并显示（标题、页码、字数）
- 构建可交互知识图谱（pyvis）
- 生成跨教材整合决策（merge/keep/remove）
- 显示压缩比不超过 30%
- RAG 问答带引用（教材、章节、页码）
- 教师反馈能修改至少一条决策
- 生成整合报告（Markdown）

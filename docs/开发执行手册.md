# 开发执行手册

本文档给主力开发 AI 使用。目标是在 5 小时内做出可提交、可部署、可演示的作品。不要追求完美系统，不要做 P1 加分项，先把 P0 简化闭环跑通。

## 1. 执行原则

1. 演示路径优先于全量能力。
2. 文档与代码并行，不要最后补。
3. 每个模块都必须有失败兜底。
4. 只做 PDF / MD / TXT，砍掉 DOCX / Excel。
5. 教师反馈做规则版，不做复杂自然语言理解。
6. RAG 必须带引用，这是硬要求。
7. 图谱必须可交互，但不追求多视图。
8. 不提交教材 PDF。

## 2. 最小产品闭环

第一版必须完成这条链路：

```text
上传 1-2 本教材
-> 解析章节
-> 切块并建立检索
-> RAG 带引用问答
-> 抽取知识点
-> 展示可交互图谱
-> 跨教材去重整合
-> 显示压缩比
-> 教师反馈修改一条决策
-> 生成整合报告
```

任何不服务这条链路的功能先不做。

## 3. 文件结构

建议只实现这些文件：


## 4. 模块一：Gradio 页面

### 必须组件

- 文件上传。
- 解析按钮。
- 文件状态表。
- 章节表。
- 图谱 HTML 区。
- 整合决策表。
- RAG 问答输入与输出。
- 教师反馈聊天框。
- 报告生成按钮和预览区。

### 页面 Tab

1. 教材管理。
2. 知识图谱。
3. 跨教材整合。
4. RAG 问答。
5. 教师反馈。
6. 整合报告。

### 验收

- `python app.py` 能启动。
- 页面可以上传文件。
- 没配置 DeepSeek 时页面不崩，提示需要 API Key。

## 5. 模块二：教材解析

### 目标

支持 PDF / Markdown / TXT，并显示章节。

### 实现范围

#### PDF

- 使用 PyMuPDF。
- 逐页读取。
- 保留页码。
- 快速模式默认只处理前 60 页或前 8 章。
- 章节正则：

```text
第[一二三四五六七八九十百零0-9]+[章节篇]
[0-9]+[.、]\s*.+
Chapter\s+[0-9]+
```

#### Markdown

- 使用 `#` / `##` 识别章节。

#### TXT

- 使用中文章节正则识别。

### 兜底

- 识别不到章节：整本作为“默认章节”。
- PDF 章节识别差：每 10 页切一个伪章节。
- 单个文件失败：标记失败，不影响其他文件。

### 验收

- 上传 1 个 PDF 可以看到章节数、字数、页码。
- 上传 1 个 MD/TXT 可以看到章节。

## 6. 模块三：RAG 问答

### 目标

优先打通问答，因为它最容易演示价值。

### 分块

- 每块约 700 字。
- overlap 100 字。
- 元数据：教材名、章节名、页码、chunk 文本。

### 检索

第一版优先用 MiniMax embo-01 API 做 embedding，避免本机下载模型和算力瓶颈。

MiniMax 请求约定：

- endpoint: `https://api.minimax.chat/v1/embeddings`
- model: `embo-01`
- 构建知识库 chunk 向量时：`type=db`
- 用户提问向量时：`type=query`

检索层先用内存向量矩阵 + cosine similarity，不引入复杂向量数据库。

如果 MiniMax API 不可用，自动回退到 sklearn TF-IDF，保证演示不中断。

### DeepSeek 回答

统一走 `llm_client.py`。

Prompt 必须要求：

- 只基于给定上下文回答。
- 不使用上下文外知识。
- 找不到则回答“当前知识库中未找到相关信息”。
- 输出中文。
- 答案后附来源引用。

### 兜底

DeepSeek 失败时，直接返回 top chunk 摘要和引用来源。

### 验收

- 输入预设问题后有答案。
- 引用表包含教材、章节、页码、相关度。
- 可查看原文 chunk。

## 7. 模块四：知识点抽取

### 目标

从章节生成节点和边。

### 实现策略

优先调用 DeepSeek。每章最多抽取 5-8 个知识点，避免慢。

输出 JSON：

```json
{
  "nodes": [
    {
      "name": "炎症",
      "definition": "机体对损伤因子的防御性反应",
      "category": "核心概念",
      "chapter": "第四章 炎症",
      "page": 78
    }
  ],
  "edges": [
    {
      "source": "炎症",
      "target": "免疫应答",
      "relation_type": "applies_to",
      "description": "炎症反应与免疫应答密切相关"
    }
  ]
}
```

关系类型至少覆盖三种：

- contains
- prerequisite
- parallel
- applies_to

### 兜底

LLM 失败时：

- 用章节标题生成一个核心节点。
- 从章节前 1000 字中抽取若干高频中文词作为节点。
- 自动生成 contains/parallel 简单关系。

### 验收

- 至少一本文档能生成节点和边。
- 关系类型不少于三种。

## 8. 模块五：图谱可视化

### 目标

图谱可交互，满足赛题验收。

### 技术

pyvis。

### 必须效果

- 可缩放。
- 可拖拽画布。
- 可拖拽节点。
- 点击或悬停看节点详情。
- 不同教材节点颜色不同。
- 高频/合并节点更大。

### 兜底

图谱 HTML 失败时，至少展示节点表和边表。

### 验收

- 页面中不是静态图片。
- 节点可拖动。
- 节点详情可见。

## 9. 模块六：跨教材整合

### 目标

2 本教材能生成去重决策和压缩比。

### 第一版算法

不要做复杂聚类。使用三层简单规则：

1. 名称规范化后相同：merge。
2. 名称包含关系或相似度高：merge 候选。
3. TF-IDF 相似度高：merge 候选。

相似度阈值建议：

```text
name_similarity >= 0.85
definition_similarity >= 0.75
```

### 决策类型

- merge：跨教材重复或高度相似。
- keep：唯一但重要。
- remove：合并后冗余短定义。

### 决策字段

- decision_id
- action
- affected_nodes
- result_node
- reason
- confidence
- status

### 压缩比

第一版允许估算，但要自洽：

```text
原始总字数 = 所有解析文本字数
整合后估算字数 = min(原始总字数 * 0.30, 合并后节点定义总字数 + 核心摘要字数)
压缩比 = 整合后估算字数 / 原始总字数
```

页面必须显示压缩比 <= 30%。

### 验收

- 上传 2 本教材后有 merge/keep/remove 决策。
- 每条决策有中文理由。
- 压缩比可见。

## 10. 模块七：教师反馈

### 目标

满足“多轮对话迭代优化”，不做复杂智能体。

### 支持指令

只做四类规则：

```text
保留 XXX
删除 XXX
不要合并 XXX 和 YYY
为什么合并 XXX
```

### 行为

- “保留”：将对应 remove 改为 keep。
- “删除”：将对应 keep 改为 remove。
- “不要合并”：将对应 merge 标记为 overridden。
- “为什么”：返回对应 reason。

### 验收

- 输入一条反馈后，决策表变化。
- 聊天历史保留在当前会话。

## 11. 模块八：整合报告

### 目标

生成赛题要求报告，不要最后手写。

### 必须包含

1. 整合概览。
2. 决策摘要。
3. 图谱统计。
4. 3-5 个典型整合案例。
5. 教学完整性说明。

### 验收

- 页面可预览。
- 写入 `report/整合报告.md`。
- 数据来自当前运行状态。

## 13. 部署要求

### 魔搭创空间

本机跑通后再部署。

需要：

- `app.py`
- `requirements.txt`
- `.env.example`
- README
- 文档目录

在魔搭环境变量中配置：

```text
DEEPSEEK_API_KEY
```

### Gradio share 保底

如果魔搭失败：

```python
demo.launch(share=True)
```

提交 share 链接作为临时公网链接。

## 14. 最低保命版本

如果时间极度紧张，保这些：

- 上传解析。
- RAG 带引用。
- 图谱有节点且可拖拽。
- 整合决策表和压缩比。
- 教师反馈能改一条决策。
- README + 三份 docs + 报告。
- 公网链接。

其他都可以简化。
